# 運用部署

## 役割

既存資産・業務の **運用・改善・自動化・モニタリング** を担う部署。

「**作った後の運用**・**毎月決まって発生する業務**・**繰り返し作業の自動化**」を扱う。

## できること（Claude への呼びかけ例）

### 月次・週次レポート

- 「今月の {対象} の成果レポート（CSV）を整理して。前月比・項目別・サイト別」
- 「整理した結果を、{担当者} に共有する用の Markdown サマリにして」
- 「先月と比べて伸びた / 落ちた項目トップ 3 をピックアップして、理由の仮説を 2 つずつ」

### KPI モニタリング

- 「今月の {KPI 名} を、目標と実績で表にまとめて」
- 「過去 3 か月のトレンドをグラフ用のデータに整理」
- 「異常値（前月比 ±30% 以上）があれば指摘して」

### 自動化スクリプト

- 「{業務名} を自動化したい。要件定義書を作って」
- 「{スクリプト名} を Python で実装して。動作確認用のサンプルデータも一緒に」
- 「このスクリプトを Windows のタスクスケジューラに登録する手順書を作って」

### 定常業務・メンテナンス

- 「毎月 1 日にやる業務をリストアップして、所要時間と担当を整理」
- 「サイトのバックアップ手順書を作って」
- 「先月期限切れになった案件・アカウントを抽出して」

### エラー対応・改善

- 「昨日動かした {スクリプト名} がエラーで止まった。ログを見て原因を特定して」
- 「この定常業務、もう少し効率化できないか改善案を 3 つ」

## 運用ルール

- ログイン情報・API キーは **このフォルダ内に直接書かない**（`.env` ファイル or Windows 資格情報マネージャ）
- スクリプトは `scripts/{用途}/{スクリプト名}.py` 形式
- 実行ログは `logs/{YYYY-MM-DD}-{スクリプト名}.log` に出力
- 大事なファイル操作（削除・上書き）は必ず確認プロンプトを挟む
- 1 つのスクリプト = 1 つの責務（あれもこれも詰め込まない）

## NG / 注意

- 認証情報を Git にコミットしない（`.gitignore` で `.env` を必ず除外）
- 本番データを直接書き換えるスクリプトは、まず **テスト環境** で動かす
- 外部 API のレート制限を超えないよう、適切な間隔を空ける
- 売上数値・個人情報の外部送信禁止

## フォルダ構成（例）

- `reports/{YYYY-MM}/` — 月次・週次レポート
- `scripts/{用途}/` — 用途別スクリプト
- `kpi/` — KPI 管理データ
- `maintenance/` — 定常業務手順書
- `logs/` — 実行ログ
- `.env.example` — 環境変数のサンプル（実際の `.env` は git 管理外）

## 部署間連携

- レポートの可視化・要約配信 → `production/` に依頼
- 顧客への運用報告 → `sales/` に共有
- 重要な業務改善方針 → `secretary/decisions/` に記録

## ナレッジソサエティ業務での想定例

最初はナレッジソサエティ様の以下の業務に使えます（運用しながらカスタマイズしてください）:

- アフィリ ASP の月次成果レポート整理（CSV → サマリ）
- ASP からの自動 DL → Google Drive 格納 → Chatwork 通知の仕組み化
- 銀行手数料等の画像差し替え運用の半自動化
- WordPress 4 サイトの定期バックアップ
- サイト KPI（PV / CV）の月次モニタリング
